# תוכנית שיפור Wandi Agent — 5 פאזות

> מטרה: לגרום ל-wandi-agent לייצר תוכן איכותי, מדויק, ומותאם ללקוח — מקצה לקצה.
> מודל: נשאר עם DictaLM, משפרים דרך prompt engineering + validation + data.

---

## פאזה 1: לתקן את השמירה לאיירטייבל (חוסם כל השאר)

**בעיה**: המודל ממציא ערכים לשדות Select (עברית/אנגלית/שילובים חדשים כל פעם).

**קבצים**: `agent.py`

**שינויים**:
1. להרחיב את מפות הנורמליזציה עם כל הווריאציות העבריות שראינו:
   - `"לא מודעים"` / `"לא מודע"` → `"Unaware"`
   - `"מודעות לבעיה"` / `"מודע לבעיה"` → `"Problem-Aware"`
   - `"פתרון"` / `"מודעות לפתרון"` / `"מודע לפתרון"` → `"Solution-Aware"`
   - `"ערכי"` / `"חינוכי"` / `"תוכן ערכי"` → `"חשיפה"`
   - `"מכירתי"` → `"מכירה"`
2. להוסיף fallback חכם: אם ערך לא מזוהה → לנקות את השדה (שלח ריק) במקום להעביר ערך שישבור

## פאזה 2: שיפור הפרומפט (ההשפעה הכי גדולה)

**בעיה**: הפרומפט שטוח — משפט אחד לכל שלב מודעות, אין דוגמאות, אין כללי CTA ברורים.

**קבצים**: `prompts.py`

**שינויים**:

### 2a. הנחיות מפורטות לכל שלב מודעות
במקום:
```
"Unaware": "תוכן ויראלי/חשיפה — לא מוכרים, רק מושכים תשומת לב"
```
להרחיב עם מידע מה-SDMF:
```
Unaware (שלב 1):
- מטרה: לעצור את הסקרול. הצופה לא מכיר אותך ולא יודע שיש לו בעיה.
- טריגר פסיכולוגי: סקרנות — לשבור תפיסת מציאות
- סוגי הוקים מתאימים: פרובוקציה, תוצאה מפתיעה, שאלה מאתגרת
- CTA: עקבו / שמרו (בלי מגנט!)
- content_type: חשיפה (תמיד!)
- פורמט מועדף: talking_head / mixed (ויראלי)
```

### 2b. דוגמאות (few-shot)
להוסיף 1-2 דוגמאות מלאות של רילס מושלם:
```
דוגמה לרילס Unaware מושלם:
{
  "hook": "עסק בלי AI ב-2025 זה כמו עסק בלי אתר ב-2005",
  "hook_type": "פרובוקציה",
  "text_on_video": null,
  "format": "talking_head",
  "content_type": "חשיפה",
  "awareness_stage": "Unaware",
  "magnet_id": null,
  ...
}
```

### 2c. כללי הוק-מגנט
הנחיה ברורה:
```
כשיוצרים רילס Solution-Aware:
- ההוק חייב ליצור סקרנות שהמגנט פותר
- הקפשן חייב לכלול CTA עם מילת הטריגר של המגנט
- דוגמה: אם המגנט הוא "אבחון עסקי" עם טריגר "אבחון" →
  הוק: "90% מבעלי העסקים לא יודעים מה באמת עוצר אותם"
  CTA: "תגיבו 'אבחון' ותקבלו אבחון עסקי בחינם"
```

### 2d. הכרחת ערכים מדויקים
להוסיף הנחיה מפורשת:
```
חשוב מאוד! השתמש אך ורק בערכים הבאים (ללא שינוי):
- hook_type: שאלה מאתגרת / מספר + הבטחה / טעות נפוצה / סוד חשיפה / זיהוי קהל / תוצאה מפתיעה / פרובוקציה
- format: talking_head / b_roll / text_animation / mixed
- content_type: חשיפה / מכירה
- awareness_stage: Unaware / Problem-Aware / Solution-Aware
אל תתרגם, אל תשנה, אל תמציא ערכים חדשים.
```

## פאזה 3: הזנת מידע חכמה יותר

**בעיה**: הוקים לא מסוננים, אין היסטוריה, אין רנדומיזציה.

**קבצים**: `airtable_client.py`, `agent.py`, `prompts.py`

**שינויים**:

### 3a. היסטוריית הוקים ללקוח
פונקציה חדשה באיירטייבל:
```python
async def get_recent_hooks(client_id: str, limit: int = 20) -> list[str]:
    """Fetch recent hooks from Content Queue for this client."""
```
להעביר למודל: "אלה הוקים שכבר יצרת ללקוח הזה — אל תחזור עליהם"

### 3b. סינון הוקים לפי שלב מודעות
במקום 10 ראשונים — לסנן לפי סוג הוק רלוונטי לשלב:
- Unaware: פרובוקציה, תוצאה מפתיעה, שאלה מאתגרת
- Problem-Aware: טעות נפוצה, זיהוי קהל
- Solution-Aware: סוד חשיפה, מספר + הבטחה

### 3c. רנדומיזציה
`random.sample(hooks, min(10, len(hooks)))` במקום `hooks[:10]`

## פאזה 4: הגדרות מודל + Retry

**בעיה**: temperature גבוה מדי, output tokens לא מספיק, אין retry.

**קבצים**: `ollama_client.py`

**שינויים**:
1. `temperature`: 0.8 → 0.65 (יותר עקבי לפורמט, עדיין יצירתי לתוכן)
2. `num_predict`: 4096 → 8192 (מספיק ל-7+ רילסים)
3. Retry: אם JSON שבור או חסרים שדות חובה → retry פעם אחת עם temperature 0.5

## פאזה 5: ולידציה אחרי יצירה

**בעיה**: אין בדיקה שהמודל החזיר מה שצריך.

**קבצים**: `agent.py`

**שינויים**:
פונקציה `_validate_reel(reel, magnets)` שבודקת:
1. כל שדות החובה קיימים ולא ריקים (hook, hook_type, format, content_type, awareness_stage)
2. Solution-Aware → חייב magnet_id (אם יש מגנטים זמינים)
3. Solution-Aware → caption מכיל trigger word של המגנט
4. Unaware → content_type == חשיפה
5. text_on_video == null כשזה hook_only, ולא null כשזה hook_with_text
6. רילסים שלא עוברים ולידציה → לוג warning + ניסיון תיקון אוטומטי (מילוי defaults)

```
פאזה 1 (30 דק) → תיקון נורמליזציה → מבטל את ה-blocker
פאזה 2 (60 דק) → שיפור פרומפט → השפעה הכי גדולה על איכות
פאזה 4 (20 דק) → הגדרות מודל → יציבות
פאזה 5 (30 דק) → ולידציה → רשת ביטחון
פאזה 3 (40 דק) → מידע חכם → שיפור לטווח ארוך

סה"כ: ~3 שעות עבודה
```

כל פאזה עצמאית — אפשר לבדוק אחרי כל אחת שהמערכת עובדת.
